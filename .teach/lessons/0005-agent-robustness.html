L0005 - Agent 健壮性处理

最大步数、超时、工具失败重试、错误回传——让 Agent 从"能跑"变成"能上线"

1. 上一课的 Agent 有什么问题？

L0004 的最小 Agent 能跑，但生产环境会遇到这些问题：

问题后果类比
LLM 陷入死循环无限调工具，烧钱while(true) 无 break
LLM 响应太慢用户等 30 秒无结果HTTP 请求无超时
工具执行失败Agent 崩溃或卡住Service 层抛异常未 catch
LLM 幻觉调不存在的工具代码报 KeyError调了不存在的 API 端点
L0004 的 Agent ≈ 没有 try-catch、没有超时、没有限流的 Controller。
能跑 ≠ 能上线。健壮性处理就是给 Agent 加护栏（Guardrails）。

2. 四道护栏

护栏 1：最大步数限制

最基础也最必须。LLM 可能反复调工具不终止，MAX_STEPS 是硬上限。

MAX_STEPS = 5

for step in range(MAX_STEPS):
    response = call_llm(messages, tools)
    if response.stop_reason != "tool_use":
        return response.content[0].text  # 正常终止
    # ... 执行工具，继续循环

# 循环走完还没 return → 超过步数限制
return f"Agent 在 {MAX_STEPS} 步内未能完成任务。"
MAX_STEPS 设多少？
简单问答：3-5 步。数据分析：10-15 步。编程 Agent：20-50 步。
宁可设大一点加超时兜底，不要设太小导致任务截断。

护栏 2：超时机制

MAX_STEPS 限制循环次数，不限制单次 LLM 调用时间。一次 LLM 调用可能卡 30 秒+。

import threading

def call_llm_with_timeout(client, messages, tools, timeout=10):
    """跨平台超时方案（Windows 兼容）"""
    result = {}

    def worker():
        result["response"] = client.messages.create(
            model="claude-sonnet-4-5-20250514",
            max_tokens=1024,
            system="你是一个助手。",
            messages=messages,
            tools=tools,
        )

    t = threading.Thread(target=worker)
    t.start()
    t.join(timeout=timeout)

    if t.is_alive():
        return None  # 超时，线程仍在后台运行
    return result.get("response")
超时 ≈ HTTP 客户端的 connectTimeout + readTimeout。
没有超时的网络调用 = 定时炸弹。RestTemplate 默认无限超时，OkuHttp 默认 10 秒——Agent 也应该有默认超时。

护栏 3：工具失败处理

工具可能抛异常（网络错误、参数不对、资源不存在）。不能让异常炸掉整个 Agent。

def execute_tool_safe(name: str, args: dict) -> str:
    """安全执行工具，失败时返回错误信息而非抛异常"""
    try:
        if name == "calculator":
            return calculator(args["expression"])
        elif name == "search":
            return search(args["query"])
        else:
            return f"错误：未知工具 '{name}'"
    except KeyError as e:
        return f"错误：工具 '{name}' 缺少必要参数 {e}"
    except Exception as e:
        return f"错误：工具 '{name}' 执行失败 - {type(e).__name__}: {e}"
工具失败时不抛异常，而是返回错误字符串给 LLM。
LLM 看到错误信息后可以：(1) 换参数重试 (2) 换工具 (3) 告诉用户"搞不定"。
比 catch 后返回固定 500 页面更灵活——LLM 能理解错误语义。

护栏 4：错误信息回传 LLM

把工具错误当作普通 tool_result 传回去：

for block in response.content:
    if block.type == "tool_use":
        result = execute_tool_safe(block.name, block.input)  # 不会抛异常
        tool_results.append({
            "type": "tool_result",
            "tool_use_id": block.id,
            "content": result,  # 正常结果或错误信息
        })

LLM 看到错误后的典型行为：

--- Step 1 ---
  calculator({'expression': '1/0'})
  结果: 错误：工具 'calculator' 执行失败 - ZeroDivisionError: division by zero

--- Step 2 ---
抱歉，1/0 是除以零，数学上无法计算。

3. 完整的健壮 Agent Loop

def robust_agent(user_input: str) -> str:
    messages = [{"role": "user", "content": user_input}]

    for step in range(MAX_STEPS):
        print(f"\n--- Step {step + 1} ---")

        # 1. 调 LLM（带超时）
        response = call_llm_with_timeout(client, messages, tools, LLM_TIMEOUT)
        if response is None:
            return "LLM 响应超时，请稍后重试。"

        # 2. 检查终止条件
        if response.stop_reason != "tool_use":
            text_parts = [b.text for b in response.content if hasattr(b, 'text')]
            return "".join(text_parts) if text_parts else "Agent 未能生成有效回答。"

        # 3. 追加 assistant 响应
        messages.append({"role": "assistant", "content": response.content})

        # 4. 安全执行工具
        tool_results = []
        for block in response.content:
            if block.type == "tool_use":
                result = execute_tool_safe(block.name, block.input)
                print(f"  {block.name}({block.input}) → {result[:80]}")
                tool_results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "content": result,
                })

        messages.append({"role": "user", "content": tool_results})

    return f"Agent 在 {MAX_STEPS} 步内未能完成任务。"
"Agent 如何处理工具调用失败？"

回答要点：
(1) 不抛异常——把错误信息作为 tool_result 回传给 LLM
(2) LLM 看到错误后自主决策：重试、换工具、告诉用户
(3) 这比 try-catch 后返回固定错误更灵活——LLM 能理解错误语义
(4) 配合 MAX_STEPS + 超时兜底，防止无限重试

4. 自测题

题目 1：工具执行失败时，最佳做法是什么？

正确。LLM 看到错误后能自主调整——重试、换工具、告诉用户。比抛异常或静默忽略都好。

题目 2：为什么需要同时设置 MAX_STEPS 和超时？

正确。MAX_STEPS 防无限循环，超时防单次调用卡住。两者防护不同故障模式。

题目 3：LLM 调了一个不存在的工具名，怎么处理？

正确。返回明确错误让 LLM 知道"这个工具不存在"，它会在下一步修正。这是 LLM 幻觉的典型场景。
