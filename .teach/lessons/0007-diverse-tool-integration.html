L2.2 · 多样化工具接入

1. 回顾：Agent = LLM + Tools + Loop

L0004 的 Agent 只有两个简单工具：calculator 和 search（假数据）。真实 Agent 需要接入各种外部系统。本课讲四类最常用的工具。

工具类型解决什么类比
搜索引擎获取实时信息调用百度/Google API
数据库查询读取结构化数据MyBatis / JPA 查询
文件系统读写本地/远程文件File I/O 操作
代码执行运行计算/脚本Runtime.exec() / Docker
Phase 1 的 Agent ≈ 只有内存运算的微服务。
本课给 Agent 加上外部依赖：HTTP Client（搜索）、DataSource（数据库）、FileSystem（文件）、Sandbox（代码执行）。
每加一类工具，Agent 的能力边界就扩大一圈。

2. 搜索引擎工具

LLM 知识有截止日期，搜索工具让 Agent 获取实时信息。

Tavily API（推荐入门）

# pip install tavily-python
import os
from tavily import TavilyClient

tavily = TavilyClient(api_key=os.environ["TAVILY_API_KEY"])

def web_search(query: str) -> str:
    """搜索互联网获取实时信息"""
    results = tavily.search(query, max_results=3)
    output = []
    for r in results["results"]:
        output.append(f"标题: {r['title']}\n来源: {r['url']}\n摘要: {r['content']}")
    return "\n\n".join(output) if output else "未找到相关结果"

print(web_search("2024 年诺贝尔物理学奖"))

Tool Schema 定义

search_tool = {
    "name": "web_search",
    "description": "搜索互联网获取最新信息。适合查询时事、新闻、最新数据。",
    "input_schema": {
        "type": "object",
        "properties": {
            "query": {
                "type": "string",
                "description": "搜索关键词"
            }
        },
        "required": ["query"]
    }
}
description 是给 LLM 看的"使用说明书"。
写得越清楚，LLM 越知道什么时候该用这个工具。例如加上"适合查询时事、新闻、最新数据"，LLM 就不会在问"1+1=?"时调搜索。
类比：接口文档写得越清楚，前端越不容易调错 API。

3. 数据库查询工具（Text-to-SQL）

让 Agent 把自然语言问题翻译成 SQL 查询。核心思路：把表结构信息放进 prompt，让 LLM 生成 SQL。

import sqlite3

# 数据库 schema 信息（给 LLM 看）
DB_SCHEMA = """
表 orders:
  - id INTEGER PRIMARY KEY
  - customer_name TEXT
  - product TEXT
  - quantity INTEGER
  - price REAL
  - order_date TEXT
"""

def text_to_sql(question: str) -> str:
    """自然语言转 SQL 并执行"""
    prompt = f"""根据以下数据库 schema，把问题转成 SQL 查询。
只返回 SQL 语句，不要其他内容。

Schema:
{DB_SCHEMA}

问题: {question}

SQL:"""

    from openai import OpenAI
    client = OpenAI()
    response = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{"role": "user", "content": prompt}],
        temperature=0,
    )
    sql = response.choices[0].message.content.strip()

    # 安全检查：只允许 SELECT
    if not sql.upper().startswith("SELECT"):
        return f"安全限制：只允许 SELECT 查询。生成的 SQL: {sql}"

    try:
        conn = sqlite3.connect("orders.db")
        cursor = conn.cursor()
        cursor.execute(sql)
        rows = cursor.fetchall()
        columns = [desc[0] for desc in cursor.description]
        conn.close()
        return f"SQL: {sql}\n列: {columns}\n结果: {rows}"
    except Exception as e:
        return f"SQL 执行错误: {e}\nSQL: {sql}"
Text-to-SQL 安全要点：
(1) 只允许 SELECT，禁止 INSERT/UPDATE/DELETE/DROP
(2) 用只读数据库连接（read-only user）
(3) 限制返回行数（加 LIMIT）
(4) 生产环境考虑用参数化查询防 SQL 注入
类比：和 MyBatis 的 ${} vs #{} 是同一个问题——永远不要直接拼接用户输入。

4. 文件系统工具

import os

def read_file(path: str) -> str:
    """读取文件内容"""
    base_dir = os.path.abspath("./workspace")
    full_path = os.path.abspath(os.path.join(base_dir, path))
    if not full_path.startswith(base_dir):
        return f"安全限制：不允许访问 {path}（路径穿越）"

    try:
        with open(full_path, encoding="utf-8") as f:
            return f.read()[:10000]
    except FileNotFoundError:
        return f"文件不存在: {path}"
    except Exception as e:
        return f"读取失败: {e}"

def list_files(directory: str = ".") -> str:
    """列出目录下的文件"""
    base_dir = os.path.abspath("./workspace")
    target = os.path.abspath(os.path.join(base_dir, directory))
    if not target.startswith(base_dir):
        return "安全限制：不允许访问该目录"

    try:
        entries = os.listdir(target)
        return "\n".join(entries) if entries else "目录为空"
    except Exception as e:
        return f"列出失败: {e}"
文件系统工具 ≈ Controller 层的文件上传下载接口。
核心安全问题是路径穿越（Path Traversal）：../../etc/passwd。
和 Web 安全的目录遍历漏洞完全一样——normalize 后检查前缀。

5. 代码执行工具

import subprocess
import tempfile

def execute_python(code: str) -> str:
    """在沙箱中执行 Python 代码"""
    blocked = ["import os", "import subprocess", "import sys",
               "open(", "exec(", "eval(", "__import__"]
    for keyword in blocked:
        if keyword in code:
            return f"安全限制：代码包含禁止的操作 '{keyword}'"

    with tempfile.NamedTemporaryFile(mode="w", suffix=".py", delete=False, encoding="utf-8") as f:
        f.write(code)
        tmp_path = f.name

    try:
        result = subprocess.run(
            ["python", tmp_path],
            capture_output=True, text=True, timeout=10,
        )
        output = result.stdout if result.returncode == 0 else f"错误:\n{result.stderr}"
        return output[:5000]
    except subprocess.TimeoutExpired:
        return "执行超时（10秒限制）"
    finally:
        os.unlink(tmp_path)
代码执行是风险最高的工具。
上面的黑名单检查只是演示。生产环境必须用Docker 容器沙箱：
- 网络隔离（无外网访问）
- 文件系统只读挂载
- 资源限制（CPU/内存/时间）
- 非 root 用户运行
参考 OpenAI Codex / Claude Code 的沙箱设计。

6. 工具设计四原则

原则为什么反例
单一职责一个工具做一件事，LLM 更容易选对"万能工具"做搜索+计算+写文件
Schema 清晰参数名和描述要自解释参数名叫 q 而非 query
错误可追溯返回具体错误信息，不吞异常catch 后返回空字符串
最小权限只开放必要权限文件工具能读写整个 / 目录
# 反面教材
bad_tool = {
    "name": "do_stuff",
    "description": "做各种事情",  # LLM 不知道什么时候该用
}

# 正面教材
good_tool = {
    "name": "web_search",
    "description": "搜索互联网获取最新信息。当用户问时事、新闻、最新数据时使用。不用于数学计算。",
    "input_schema": {
        "type": "object",
        "properties": {
            "query": {"type": "string", "description": "搜索关键词，1-5 个词最佳"}
        },
        "required": ["query"]
    }
}
"设计 Agent 工具时有哪些原则？怎么防止工具被滥用？"

回答要点：
(1) 单一职责——一个工具做一件事，description 写清适用场景
(2) Schema 清晰——参数名自解释，type 和 required 明确
(3) 最小权限——只读连接、路径限制、沙箱隔离
(4) 错误可追溯——返回具体错误信息给 LLM，不吞异常
(5) 代码执行必须沙箱化（Docker），SQL 工具只允许 SELECT

7. 统一工具注册

tools = [search_tool, sql_tool, read_file_tool, list_files_tool, execute_python_tool]

TOOL_MAP = {
    "web_search": lambda args: web_search(args["query"]),
    "text_to_sql": lambda args: text_to_sql(args["question"]),
    "read_file": lambda args: read_file(args["path"]),
    "list_files": lambda args: list_files(args.get("directory", ".")),
    "execute_python": lambda args: execute_python(args["code"]),
}

def execute_tool(name: str, args: dict) -> str:
    handler = TOOL_MAP.get(name)
    if not handler:
        return f"未知工具: {name}"
    try:
        return handler(args)
    except Exception as e:
        return f"工具 '{name}' 执行失败: {type(e).__name__}: {e}"
工具注册 ≈ Spring 的 @RequestMapping 路由表。
TOOL_MAP 是 DispatcherServlet（路由到 handler），LLM 是前端（发请求），handler 是 Controller 方法。

8. 自测题

题目 1：Tool Schema 的 description 最重要的作用是什么？

正确。description 是 LLM 选择工具的主要依据。写得越精准，LLM 越不容易调错。

题目 2：Text-to-SQL 工具最重要的安全措施是什么？

正确。最核心是权限控制——只允许 SELECT + 只读连接。LLM 生成的 SQL 不可信，必须在数据库层面做权限兜底。

题目 3：为什么代码执行工具必须用沙箱？

正确。LLM 可能生成恶意代码。黑名单不够——生产环境必须用 Docker 隔离：网络隔离、文件系统只读、资源限制、非 root 运行。
