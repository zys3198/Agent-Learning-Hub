L0002 - LLM API 基础调用

从 curl 到 Python SDK：理解 messages 结构和核心参数

1. 先搞清一件事：你调的不是"模型"，是"API"

LLM 不在你本地运行。你的代码做的事情是：构造一个 HTTP 请求，发给远程服务器，拿到 JSON 响应。

调 LLM API ≈ 调一个远程 REST 接口。
URL 就是 endpoint，messages 就是请求体，响应里的 content 就是返回值。
client.chat.completions.create() ≈ RestTemplate.postForObject()，SDK 只是把 HTTP 细节封装了。

所以学习顺序是：

  1. 先用 curl 直接调（理解底层 HTTP）
  2. 再用 Python SDK 调（工程实践中用这个）

2. curl 方式：看到裸的 HTTP 请求

# OpenAI API — 最基础的聊天请求
curl https://api.openai.com/v1/chat/completions \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $OPENAI_API_KEY" \
  -d '{
    "model": "gpt-4o-mini",
    "messages": [
      {"role": "system", "content": "你是一个有帮助的助手。"},
      {"role": "user", "content": "什么是 Agent？一句话回答。"}
    ],
    "temperature": 0.7,
    "max_tokens": 100
  }'

响应结构：

{
  "choices": [{
    "message": {
      "role": "assistant",
      "content": "Agent 是能自主感知环境、做决策、执行动作的 AI 程序。"
    },
    "finish_reason": "stop"
  }],
  "usage": {
    "prompt_tokens": 28,
    "completion_tokens": 18,
    "total_tokens": 46
  }
}
核心数据结构就是 messages 数组：每条消息有 role 和 content。
你传进去的 messages 越长，token 消耗越多，费用越高。

3. messages 结构：四种角色

角色作用Java 类比
system设定 AI 的行为规则和人设Filter/Interceptor 中的初始化配置
user用户的输入HTTP Request Body
assistantAI 的回复HTTP Response Body
tool工具调用的返回结果（下节课详解）RPC 的返回值

对话就是不断往 messages 数组追加消息：

# 第一轮
messages = [
    {"role": "system", "content": "你是 Java 专家。"},
    {"role": "user", "content": "什么是 Spring IOC？"},
]
# → AI 回复后，追加 assistant 消息

# 第二轮（带上历史）
messages = [
    {"role": "system", "content": "你是 Java 专家。"},
    {"role": "user", "content": "什么是 Spring IOC？"},
    {"role": "assistant", "content": "IOC 是控制反转..."},
    {"role": "user", "content": "和 DI 有什么区别？"},  # 新问题
]
messages 数组 ≈ HTTP Session 中的聊天历史。
每次请求都是无状态的——服务器不记得你，你每次要把完整历史都传过去。
这就是为什么上下文长度（context window）那么重要——它限制了 messages 数组的总长度。

4. Python SDK 调用

4.1 OpenAI SDK

from openai import OpenAI

client = OpenAI()  # 自动读 OPENAI_API_KEY 环境变量

response = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[
        {"role": "system", "content": "你是一个有帮助的助手。"},
        {"role": "user", "content": "用一句话解释什么是 API。"},
    ],
    temperature=0.7,
    max_tokens=100,
)

# 访问响应
print(response.choices[0].message.content)
print(f"Token 用量: {response.usage.prompt_tokens} + {response.usage.completion_tokens}")

4.2 Anthropic SDK（Claude）

from anthropic import Anthropic

client = Anthropic()  # 自动读 ANTHROPIC_API_KEY 环境变量

response = client.messages.create(
    model="claude-sonnet-4-5-20250514",
    max_tokens=100,      # Claude 必填！
    system="你是一个有帮助的助手。",  # system 不在 messages 里！
    messages=[
        {"role": "user", "content": "用一句话解释什么是 API。"},
    ],
    temperature=0.7,
)

# 访问响应 — 注意是 .content[0].text，不是 .content
print(response.content[0].text)
print(f"Token 用量: {response.usage.input_tokens} + {response.usage.output_tokens}")

5. 两家 API 关键差异

差异点OpenAIAnthropic (Claude)
system 消息在 messages 数组内独立的 system 参数
max_tokens可选（推荐用 max_completion_tokens）必填
响应内容.choices[0].message.content.content[0].text
Token 用量字段prompt_tokens / completion_tokensinput_tokens / output_tokens
结束原因finish_reason: "stop"stop_reason: "end_turn"
第一条消息可以是 system必须是 user
"OpenAI 和 Anthropic 的 API 有什么区别？"

回答要点：
(1) system prompt 位置不同——OpenAI 放 messages 里，Anthropic 是独立参数
(2) Anthropic 的 max_tokens 是必填参数，OpenAI 可选
(3) 响应结构不同——OpenAI 用 choices 数组，Anthropic 用 content 数组
(4) 本质相同：都是 HTTP 请求，都是 messages 进、文本出

6. 核心参数怎么调

参数作用什么时候调
temperature 控制随机性。0 = 确定性输出，1 = 正常随机，2 = 非常随机 需要确定性（代码生成、数据提取）→ 0；需要创意（写作、头脑风暴）→ 0.7-1.0
max_tokens 限制回复的最大长度 需要短回答（分类、提取）→ 100-500；需要长回答（文章、代码）→ 2000-4096
model 选择模型能力层级 简单任务用小模型（快+便宜），复杂推理用大模型
temperature ≈ 数据库隔离级别。
temperature=0 ≈ SERIALIZABLE（完全确定性）
temperature=1 ≈ READ COMMITTED（正常并发）
不是越高越好——确定性场景用 0，创意场景用 0.7-1.0。

7. 自测题

题目 1：OpenAI 的 system prompt 放在哪里？

正确。OpenAI 把 system 消息放在 messages 数组内，和 user/assistant 消息同级。Anthropic 则用独立的 system 参数。

题目 2：以下哪个参数在 Anthropic API 中是必填的？

正确。Anthropic 的 max_tokens 是必填参数，不传会报错。OpenAI 的 max_tokens 则是可选的。（model 也是必填，但题目问的是"容易忽略的必填参数"——max_tokens 是最常见的坑。）

题目 3：如果需要 LLM 做数据提取（从文本中提取结构化信息），temperature 应该设多少？

正确。数据提取需要确定性输出——同样的输入应该得到同样的结果。temperature=0 让 LLM 总选概率最高的 token。
