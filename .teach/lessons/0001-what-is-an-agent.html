L0001 - Agent 到底是什么？

Chatbot vs Workflow vs Agent vs Multi-Agent · 五种 Agent 模式 · 何时该用 Agent

1. 四种 LLM 应用形态

LLM 应用不是只有"聊天机器人"。按决策权从人到机器排列：

类型 决策者 执行流程 Java 类比 典型例子
Chatbot 人（用户主导） 一问一答，无工具，无多步推理 Controller 返回 JSON，不调 Service ChatGPT 基础对话
Workflow 代码（预定义 DAG） 固定步骤：A → B → C，条件分支写在代码里 Spring Batch / Airflow DAG 发票 OCR → 数据校验 → 入库
Agent LLM（动态决策） LLM 自己决定调什么工具、调几次、什么时候停 DispatcherServlet + 反射调用，但由 LLM 选方法 Claude Code、Cursor Agent
Multi-Agent 多个 LLM 协作 Supervisor/Router 编排多个专职 Agent 微服务 + API Gateway DeerFlow、MetaGPT
一图理解：
Chatbot ≈ @RestController，直接返回，无业务逻辑。
Workflow ≈ Spring Batch Job，Step 写死在配置里。
Agent ≈ 一个带反射的 Dispatcher——method = llm.decide(context)，method.invoke()。
Multi-Agent ≈ 微服务集群，每个 Agent 是一个 Service，通过 Gateway 编排。
核心区分维度只有一个：谁决定下一步做什么？
Chatbot → 人决定。
Workflow → 代码决定（if/else/switch）。
Agent → LLM 决定。
面试中只要答清这一条，就过了及格线。

2. Agent 的核心循环：Observe → Think → Act

所有 Agent 都跑同一个循环，来源是 2022 年的 ReAct 论文（Reasoning + Acting）：

伪代码：

messages = [{"role": "user", "content": user_input}]

while steps < MAX_STEPS:
    # 1. Observe + Think: 把当前状态发给 LLM
    response = llm.chat(messages, tools=available_tools)

    # 2. 判断：LLM 决定是调工具还是直接回答
    if response.has_tool_calls:
        # 3. Act: 执行工具，把结果追加回 messages
        tool_results = execute_tools(response.tool_calls)
        messages.append(response)           # LLM 的 tool_call 决策
        messages.append(tool_results)       # 工具执行结果
        steps += 1
    else:
        return response.content  # 最终答案，循环结束
Agent Loop ≈ 事件循环（Event Loop）。
Observe ≈ 从事件队列取事件。
Think ≈ 事件处理器的决策逻辑。
Act ≈ 执行副作用（DB 写入、RPC 调用等）。
循环直到事件队列为空——Agent 里是直到 LLM 不再调工具。

3. Anthropic 五种模式：从简单到复杂

来源：Building Effective Agents（Anthropic 官方工程博客，2024 年 12 月）。核心观点：从简单开始，确认不够再升级。

模式 1: Augmented LLM（增强 LLM）
LLM + 工具，但只有一步，不循环。用户问 → LLM 调工具 → 返回。
类比：带 @Autowired 的 Controller——可以调 Service，但流程不循环。
模式 2: Prompt Chaining（提示链）
A 的输出作为 B 的输入，串行执行。每步之间可以加检查点（guard）。
类比：责任链模式（Filter Chain）——每个 Filter 处理完传给下一个。
模式 3: Routing（路由）
先分类输入，再路由到不同处理分支。类似 Controller 的 URL 路由。
类比：@RequestMapping——根据请求路径分发到不同 handler。
模式 4: Parallelization（并行）
多个 LLM 调用同时执行，结果汇总。分 sectioning（分段处理）和 voting（投票）。
类比：CompletableFuture.allOf()——并行调多个 Service，汇总结果。
模式 5: Orchestrator-Workers（编排者-执行者）
一个 LLM 做规划，动态分解子任务，分发给多个 LLM 执行，汇总结果。
类比：Master-Worker 模式——Master 分配任务，Worker 执行，结果汇总。
模式 6: Autonomous Agent（自主 Agent）
LLM 自己规划、执行、反思，人类只在必要时介入。最灵活，但也最难控制。
类比：后台守护线程——有自主逻辑，只在异常时通知人。

复杂度递增：

"Agent 有哪些常见模式？"

不要只说 "Agent 就是 LLM 加工具"。要说：
(1) Anthropic 定义了 6 种模式，从简单到复杂
(2) 不是所有 Agent 都需要自主循环——简单的 Routing/Prompt Chaining 也是 Agent
(3) 实际项目通常混用——关键路径用 Workflow，模糊地带用 Agent

4. OpenAI 的三个组件：Model + Tools + Instructions

来源：A Practical Guide to Building AI Agents（OpenAI 官方，2025 年 1 月）。核心框架：

组件作用Java 类比
ModelLLM 大脑——推理、决策、生成Spring Bean 的业务逻辑
ToolsAgent 的"手"——搜索、计算、读写文件、调 APIService 层的依赖注入
Instructions行为规则——System Prompt + 约束 + 护栏配置文件 + AOP 拦截器

Guardrails（护栏）是必须的：

  • 输出校验：LLM 输出格式不对就重试（类比 @Valid 参数校验）
  • 工具权限：只暴露必要的工具（类比 Spring Security 方法鉴权）
  • 人工确认：高风险操作前暂停等确认（类比审批流）
  • 成本上限：限制最大步数和 token 预算（类比 Rate Limiter）

5. 什么时候不该用 Agent？

Agent 引入三类风险：不确定性（LLM 可能选错工具）、成本（每次 LLM 调用都要钱）、延迟（多步循环比单次调用慢）。

场景为什么不用 Agent该用什么
任务流程固定LLM 决策多余且不可靠Workflow（Spring Batch）
任务可预测脚本足够，Agent 增加延迟和成本普通脚本 / REST API
对准确性要求极高LLM 有幻觉风险规则引擎 + 人工审核
成本敏感每次 LLM 调用都有 token 费用缓存 + 模板
"你的场景为什么需要 Agent，而不是 Workflow？"

回答模板："当任务步骤无法预定义、需要根据中间结果动态决策时，才需要 Agent。如果流程固定、输入输出可枚举，Workflow 更可靠且成本更低。实际项目中通常混用——关键路径用 Workflow 保可靠性，模糊地带用 Agent 处理意外。"

6. 自测题

题目 1：以下哪个场景最适合用 Agent？

正确。B 需要动态决定搜什么、搜几个源、怎么综合——这是 Agent 的强项。A/C/D 流程固定，用 Workflow 或脚本更可靠。

题目 2：Agent Loop 里 "Think" 阶段做什么？

正确。"Think" 是 LLM 的推理/决策阶段：分析当前状态，决定是调工具还是直接给出最终答案。

题目 3：以下哪个不是 Agent 必须的组件？

正确。向量数据库是 RAG 的组件，不是 Agent 必需的。最简 Agent = LLM + 工具 + 循环。

题目 4：Anthropic 五种模式中，Prompt Chaining 最接近哪个 Java 模式？

正确。Prompt Chaining 是串行链——A 的输出给 B，B 的输出给 C，中间可以加检查点。和 Servlet Filter Chain / Spring Interceptor Chain 是同一个思路。
