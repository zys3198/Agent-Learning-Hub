L0006 - RAG Pipeline 全流程

让 Agent 拥有知识库：文档加载 → 分块 → 向量化 → 检索 → 生成带引用的回答

L2.1 · RAG Pipeline 全流程

1. 为什么需要 RAG？

Phase 1 的 Agent 只有 LLM 内置知识（训练数据截止日期之前的），加上你给的 system prompt。但它不知道你公司的内部文档、最新新闻、私有数据库。

RAG（Retrieval-Augmented Generation）解决的就是这个问题：先从知识库中检索相关内容，再让 LLM 基于检索结果回答。

RAG ≈ 带资料的开卷考试。
没有 RAG：LLM 凭记忆回答（闭卷）。
有 RAG：LLM 先查资料，再基于资料回答（开卷）。
类比后端：RAG ≈ 先查 Redis 缓存再拼响应，而非只用代码里的硬编码常量。
RAG 不是让 LLM "学习"新知识。
LLM 的参数没变。RAG 是把检索到的文本片段塞进 prompt，让 LLM 基于这些片段回答。
类比：不是让员工背资料，而是让员工带着资料上考场。

2. RAG Pipeline 七步全景

前 4 步是离线建库（做一次），后 3 步是在线查询（每次提问都走）。

离线建库 ≈ 建 Elasticsearch 索引（倒排索引、分词、存储）。
在线查询 ≈ 用户搜索（query → 分词 → 匹配 → 返回结果）。
RAG 只是把"关键词匹配"换成了"语义向量匹配"。

3. Step 1-2：文档加载与分块

文档加载

# pip install pypdf beautifulsoup4
from pypdf import PdfReader
from bs4 import BeautifulSoup

def load_pdf(path: str) -> str:
    reader = PdfReader(path)
    return "\n".join(page.extract_text() for page in reader.pages)

def load_html(path: str) -> str:
    with open(path, encoding="utf-8") as f:
        soup = BeautifulSoup(f, "html.parser")
    return soup.get_text()

def load_markdown(path: str) -> str:
    with open(path, encoding="utf-8") as f:
        return f.read()

分块策略

LLM 上下文窗口有限（4K-200K tokens），不能把整本 PDF 塞进去。需要切成小块。

策略原理优点缺点
固定长度每 500 字符切一刀，重叠 50 字符简单、可控可能切断语义
递归字符按 \n\n → \n → 。→ 空格 递归切尊重段落边界chunk 大小不均
语义分块用 embedding 相似度找断点语义最连贯计算成本高
# 最常用：递归字符分块
from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,       # 每个 chunk 最大 500 字符
    chunk_overlap=50,     # 相邻 chunk 重叠 50 字符（防切断）
    separators=["\n\n", "\n", "。", "，", " "],
)

chunks = splitter.split_text(long_text)
print(f"分成 {len(chunks)} 个 chunk，平均长度 {sum(len(c) for c in chunks) // len(chunks)}")
chunk_size 设多少？
太小（< 200）：上下文不够，LLM 看不出完整含义。
太大（> 1000）：塞进太多无关内容，LLM 被噪声干扰，且浪费 token。
经验值：300-500 字符（中文）适合大多数场景。根据回答质量调优。

4. Step 3-4：向量化和存储

什么是 Embedding？

Embedding 是把文本映射到一个高维向量（如 1536 维）。语义相近的文本，向量距离近。

"猫吃鱼"     → [0.12, -0.34, 0.56, ..., 0.78]  # 1536 维
"猫咪吃小鱼"  → [0.11, -0.33, 0.55, ..., 0.77]  # 很接近
"Java 多线程" → [0.89, 0.12, -0.45, ..., 0.23]  # 很远
Embedding ≈ 给每段文本算一个"语义指纹"。
就像数据库给每条记录建索引——只不过传统索引按关键词，向量索引按语义。
类比 MySQL：FULLTEXT INDEX 用关键词匹配，VECTOR INDEX 用余弦相似度匹配。

代码：生成 Embedding + 存入 FAISS

# pip install faiss-cpu openai
import faiss
import numpy as np
from openai import OpenAI

client = OpenAI()

def get_embeddings(texts: list[str]) -> np.ndarray:
    """批量获取 embedding 向量"""
    response = client.embeddings.create(
        model="text-embedding-3-small",
        input=texts,
    )
    vectors = [item.embedding for item in response.data]
    return np.array(vectors, dtype="float32")

# 生成 embeddings
embeddings = get_embeddings(chunks)

# 建立向量索引
dimension = embeddings.shape[1]  # 1536
index = faiss.IndexFlatIP(dimension)  # 内积相似度（需先归一化）
faiss.normalize_L2(embeddings)        # L2 归一化后内积 = 余弦相似度
index.add(embeddings)

print(f"索引了 {index.ntotal} 个 chunk")
向量库类型适用场景类比
FAISS本地库原型验证、小数据集SQLite
Chroma本地服务开发调试、中等数据集H2 Database
Pinecone云端托管生产环境、大规模RDS
pgvectorPostgreSQL 扩展已有 PG 基础设施MySQL + 全文索引

5. Step 5：检索

def retrieve(question: str, index, chunks: list[str], top_k: int = 3) -> list[str]:
    """用问题检索最相关的 top_k 个 chunk"""
    # 问题也变成向量
    q_embedding = get_embeddings([question])
    faiss.normalize_L2(q_embedding)

    # 在向量库中找最相似的
    scores, indices = index.search(q_embedding, top_k)

    results = []
    for score, idx in zip(scores[0], indices[0]):
        results.append({
            "chunk": chunks[idx],
            "score": float(score),
            "index": int(idx),
        })
    return results

# 使用
results = retrieve("什么是 RAG？", index, chunks, top_k=3)
for r in results:
    print(f"[相似度: {r['score']:.3f}] {r['chunk'][:100]}...")
top_k 设多少？
太少（1-2）：可能漏掉关键信息。
太多（> 10）：引入噪声，LLM 被无关内容干扰。
经验值：3-5 适合大多数问答场景。重要场景可到 10-20 配合 rerank。

6. Step 6：重排（Rerank，可选但推荐）

向量检索用双塔模型（query 和 chunk 各自编码再算相似度），速度快但精度有限。Rerank 用 cross-encoder（query 和 chunk 一起编码），更准但更慢。

流程：检索 Top-20 → Rerank 精排 → 取 Top-3

向量检索：快但粗。先筛出候选集。
Rerank：  慢但精。从候选集里挑最好的。
向量检索 ≈ Elasticsearch 粗筛（BM25 从百万文档筛出 Top 100）。
Rerank ≈ 业务层精排（从 Top 100 按业务规则排最优的 10 条）。
两阶段是搜索系统的经典模式。

7. Step 7：生成带引用的回答

def answer_with_citation(question: str, retrieved_chunks: list[dict]) -> str:
    """基于检索结果生成带引用的回答"""
    context_parts = []
    for i, r in enumerate(retrieved_chunks, 1):
        context_parts.append(f"[来源 {i}] {r['chunk']}")
    context = "\n\n".join(context_parts)

    prompt = f"""基于以下资料回答问题。如果资料中没有答案，说"资料中未找到相关信息"。
回答时标注来源编号，如 [来源 1]。

资料：
{context}

问题：{question}"""

    response = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{"role": "user", "content": prompt}],
        temperature=0,
    )
    return response.choices[0].message.content

# 完整 RAG 查询
question = "什么是 RAG？"
results = retrieve(question, index, chunks, top_k=3)
answer = answer_with_citation(question, results)
print(answer)

回答要点：
(1) 七步流程：加载 → 分块 → 向量化 → 存储 → 检索 → 重排 → 生成
(2) 离线建库（1-4）vs 在线查询（5-7）
(3) chunk_size 太小丢失上下文，太大引入噪声，300-500 是经验值
(4) 向量检索快但粗，rerank 慢但精，两阶段是经典模式
(5) temperature=0 减少 RAG 场景的幻觉

9. 自测题

题目 1：RAG 中 LLM 的参数发生变化了吗？

正确。RAG 不改变 LLM 参数。它把检索到的文本片段作为 context 放进 prompt，LLM 基于这些内容生成回答。类比：开卷考试不改变你的大脑，只是让你可以翻书。

题目 2：chunk_size 设太小（如 50 字符）会导致什么问题？

正确。50 字符可能只有一句话的片段，缺少上下文。LLM 看到的 chunk 不完整，回答质量下降。chunk_size 的调优是 RAG 效果的关键杠杆之一。

题目 3：为什么推荐向量检索 + Rerank 两阶段？

正确。向量检索从百万 chunk 中快速筛出 Top-20（毫秒级），Rerank 再精排到 Top-3（百毫秒级）。速度和精度兼顾——和搜索引擎的粗排+精排是同一个思路。
