L0003 - Tool Schema 与结构化输出

让 LLM 不只"说话"，还能"调函数"：理解 Tool Call 的完整数据流

1. 为什么需要 Tool Call？

上节课 LLM 只能返回文本。但真实场景中，你需要它查数据库、调 API、执行计算、读写文件。LLM 自己做不了这些——它能做的是告诉你它想调什么函数、传什么参数。你来执行，把结果喂回去。

Tool Call ≈ RPC 调用。
LLM 是客户端（发出调用请求），你的代码是服务端（真正执行）。
LLM 不直接执行——它只输出 {"function": "get_weather", "args": {"city": "北京"}}。
就像前端调后端 API：前端不连数据库，它只发 HTTP 请求。

2. Tool Schema：告诉 LLM 有什么工具

Tool Schema 是一个 JSON 描述：工具叫什么、做什么、参数什么类型。底层就是 JSON Schema。

OpenAI 格式

tools = [
    {
        "type": "function",              # 固定值
        "name": "get_weather",           # 函数名
        "description": "查询指定城市的当前天气",  # 告诉 LLM 什么时候该用
        "parameters": {                  # ← 注意：叫 parameters
            "type": "object",
            "properties": {
                "city": {
                    "type": "string",
                    "description": "城市名，如'北京'"
                },
                "unit": {
                    "type": "string",
                    "enum": ["celsius", "fahrenheit"]
                }
            },
            "required": ["city"]
        }
    }
]

Anthropic 格式

tools = [
    {
        "name": "get_weather",
        "description": "查询指定城市的当前天气",
        "input_schema": {                # ← 注意：叫 input_schema（不是 parameters）
            "type": "object",
            "properties": {
                "city": {
                    "type": "string",
                    "description": "城市名，如'北京'"
                },
                "unit": {
                    "type": "string",
                    "enum": ["celsius", "fahrenheit"]
                }
            },
            "required": ["city"]
        }
    }
]
Tool Schema ≈ 接口契约（像 Swagger/OpenAPI spec）。
name ≈ 接口路径，description ≈ 接口描述，parameters/input_schema ≈ 请求参数 schema。
没有这个契约，LLM 不知道能调什么——就像没有 Swagger 文档，前端不知道有哪些 API。

3. 发送请求 + 解析 Tool Call 响应

OpenAI

from openai import OpenAI
import json

client = OpenAI()
response = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[{"role": "user", "content": "北京今天天气怎么样？"}],
    tools=tools,
)

message = response.choices[0].message

if message.tool_calls:
    for tc in message.tool_calls:
        print(f"函数名: {tc.function.name}")
        print(f"参数: {json.loads(tc.function.arguments)}")  # ← arguments 是字符串！
        print(f"调用 ID: {tc.id}")

Anthropic

from anthropic import Anthropic

client = Anthropic()
response = client.messages.create(
    model="claude-sonnet-4-5-20250514",
    max_tokens=1024,
    messages=[{"role": "user", "content": "北京今天天气怎么样？"}],
    tools=tools,
)

if response.stop_reason == "tool_use":
    for block in response.content:
        if block.type == "tool_use":
            print(f"函数名: {block.name}")
            print(f"参数: {block.input}")  # ← 已经是 dict，不需要 json.loads
            print(f"调用 ID: {block.id}")

4. 三大坑：OpenAI vs Anthropic Tool Call 差异

差异点OpenAIAnthropic
Schema 字段名parametersinput_schema
需要 type 字段？是："type": "function"否
参数数据类型arguments 是 JSON 字符串（需 json.loads）input 是 已解析 dict
响应位置.choices[0].message.tool_calls[].content[] 中 type="tool_use" 的 block
停止原因finish_reason == "tool_calls"stop_reason == "tool_use"
结果回传 rolerole: "tool"role: "user" + tool_result block
最常踩的坑：OpenAI arguments 是字符串，Anthropic input 是 dict。
OpenAI：args = json.loads(tc.function.arguments)
Anthropic：args = block.input（直接用）
忘了 json.loads 会导致参数解析失败——Tool Call 最常见的 Bug。

5. 回传 Tool 执行结果

OpenAI

# 1. 追加 LLM 的 tool_call 决策到 messages
messages.append(message)

# 2. 追加工具执行结果
messages.append({
    "role": "tool",                          # role 是 "tool"
    "tool_call_id": tc.id,                   # 匹配调用 ID
    "content": "北京：晴，25°C"
})

# 3. 再调 LLM，让它根据结果生成回答
response2 = client.chat.completions.create(
    model="gpt-4o-mini", messages=messages, tools=tools,
)
print(response2.choices[0].message.content)

Anthropic

# 1. 追加 LLM 的 tool_use 响应
messages.append({"role": "assistant", "content": response.content})

# 2. 追加工具执行结果（注意 role 是 "user"！）
messages.append({
    "role": "user",                          # role 是 "user"！不是 "tool"
    "content": [{
        "type": "tool_result",
        "tool_use_id": block.id,             # 匹配调用 ID
        "content": "北京：晴，25°C"
    }]
})

# 3. 再调 LLM
response2 = client.messages.create(
    model="claude-sonnet-4-5-20250514",
    max_tokens=1024, messages=messages, tools=tools,
)
print(response2.content[0].text)
"Tool Call 的完整流程是什么？"

回答要点：
(1) 定义 Tool Schema（函数名、描述、参数类型）——相当于接口契约
(2) 发送 messages + tools 给 LLM
(3) LLM 返回 tool_call（函数名 + 参数）——LLM 不执行，只表达意图
(4) 你的代码执行函数，把结果追加回 messages
(5) 再调 LLM，让它根据结果生成最终回答
(6) Agent Loop 就是反复做 (2)-(5)

6. 自测题

题目 1：LLM 能直接执行函数吗？

正确。LLM 只输出结构化的"调用意图"，你的代码负责真正执行。

题目 2：OpenAI 的 tool_call 响应中，arguments 是什么类型？

正确。OpenAI arguments 是 JSON 字符串，必须 json.loads()。Anthropic 的 input 则是已解析 dict。

题目 3：Anthropic 回传 tool_result 时，消息的 role 应该是什么？

正确。Anthropic 的 tool_result 放在 role="user" 消息内。OpenAI 用独立的 role="tool"。
