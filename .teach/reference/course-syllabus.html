P1 基础概念与最小 Agent Loop（Stage 0-1）

Stage 0: 理解 Agent 是什么

目标：能清晰区分 chatbot / workflow / agent / multi-agent，说清什么时候该用 agent。
依赖：无（起点）
课时：3-4h

L0.1 概念辨析

  • Chatbot：一问一答，无工具，无多步推理
  • Workflow：预定义 DAG，决策权在代码（类比 Spring Batch / Airflow DAG）
  • Agent：LLM 自主决策下一步，循环 until done（类比事件驱动架构中的消费者）
  • Multi-Agent：多个 Agent 协作，常见角色 planner / executor / reviewer / critic / router
  • 什么时候不该用 Agent：任务可预测、流程稳定、普通脚本能解决
"请解释 chatbot、workflow、agent、multi-agent 的区别。什么场景下你会选择 agent 而不是 workflow？"

L0.2 Agent 核心循环

  • Observe（感知环境：接收用户输入 + 工具返回）
  • Think（LLM 推理：分析当前状态，决定下一步）
  • Act（执行动作：调用工具 / 返回结果 / 请求更多信息）
  • 循环 until 满足终止条件（最终答案 / 最大步数 / 超时）

L0.3 必读文章精读

  • Anthropic: Building effective agents — 5 种 agent 模式、augmented LLM 概念、从简单到复杂
  • OpenAI: A practical guide to building agents — agent 三层定义、编排模式
写一页短笔记（500字以内），回答："我的场景为什么需要 agent，而不是普通 workflow？" 包含具体业务场景。

Stage 1: 构建最小 Agent Loop

目标：手写 50-150 行最小 agent，能选择工具、执行工具、返回答案。
依赖：Stage 0
课时：5-6h

L1.1 LLM API 基础调用

  • 用 Python 调用 OpenAI / Claude API 完成普通对话
  • messages 结构：system / user / assistant / tool 角色
  • 参数：temperature / max_tokens / model 选择

L1.2 结构化输出与 Tool Schema

  • 让模型输出结构化 JSON（response_format / JSON mode）
  • Tool Schema 设计：name / description / parameters（JSON Schema）
  • 对比 OpenAI / Claude / Gemini 三家 tool call 格式差异

L1.3 实现最小 Agent Loop

  • 定义 2-3 个工具函数（search / calculator / read_file）
  • 解析 model 的 tool_call 响应
  • 执行工具，结果追加到 messages
  • 循环 until stop_reason == "end_turn"
"请在白板上画出 agent loop 的流程图，标注每一步的数据结构变化。"

L1.4 健壮性处理

  • 最大步数限制（防止无限循环）
  • 超时机制
  • 工具失败的重试与降级
  • 错误信息如何回传给 LLM
OpenAI Function Calling · Claude Tool Use · Gemini Function Calling
提交可运行的 Python 脚本（50-150 行），calculator + search 两个工具，有最大步数限制和错误处理。

P2 RAG、工具与记忆（Stage 2）

目标：实现资料研究助手，能搜索、检索、总结并输出引用链接。
依赖：Stage 1 agent loop 能跑通
课时：10-12h

L2.1 RAG Pipeline 全流程

  • 文档加载：PDF / HTML / Markdown -> 统一文本
  • Chunking 分块：固定长度 / 语义分块 / 递归分块
  • Embedding 向量化：文本 -> 高维向量
  • 向量存储：FAISS / Chroma / Pinecone / pgvector
  • Retrieve 检索：余弦相似度 / ANN，Top-K 参数
  • Rerank 重排：cross-encoder 精排
  • Answer with Citations：检索结果进 prompt，标注来源
"请描述 RAG 的完整 pipeline。chunking 策略对最终回答质量有什么影响？"

L2.2 多样化工具接入

  • 搜索引擎工具（Tavily / SerpAPI）
  • 数据库工具（Text-to-SQL）
  • 文件系统工具（读/写/搜索）
  • 代码执行工具（sandboxed Python / Docker）
  • 工具设计原则：单一职责、schema 清晰、错误可追溯

L2.3 记忆系统设计

  • 短期记忆：当前会话 messages，随上下文窗口滑动
  • 会话记忆：跨 turn 对话摘要
  • 长期记忆：跨会话持久化（用户偏好、知识积累）
  • 类比：短期 = request scope，会话 = session scope，长期 = DB

L2.4 常见问题处理

  • 工具失败、空结果、重复调用的降级
  • 幻觉引用的检测与缓解
  • 回答中给出来源/证据
LlamaIndex Agents · LangChain · Gemini Code Execution · MCP · mem0 · Letta
RAG 资料 agent：输入主题 -> 搜索 + 检索 -> 筛选 -> 总结 -> 输出引用链接。chunking 策略可配置。

P3 现代 Agent Harness 架构（Stage 3）

目标：能画出 coding agent harness 架构图，解释工具注册、权限、上下文压缩、subagents。
依赖：Stage 1-2
课时：10-14h

L3.1 Harness 概念

  • Harness = Agent 运行时框架，类比 Spring 容器
  • 核心组件：Agent Loop / Tool Registry / Permission Gate / Session Store / Context Compaction / Logger
  • 裸 agent loop vs harness 的差异

L3.2 Claude Code 架构拆解

  • CLI 交互层：用户输入 -> system prompt -> agent loop -> 输出
  • 工具协议：Bash / Read / Write / Edit / Glob / Grep 等，权限等级
  • 权限系统：自动允许 / 用户确认 / 拒绝三级
  • 上下文压缩：长对话 compaction（摘要 + 保留关键信息）
  • Subagents：任务拆分、上下文隔离
  • Hooks：拦截、校验、扩展 agent 行为
  • MCP 集成：外部工具/数据源标准化连接

P4 协议与 Skills（Stage 5）

目标：说清 MCP / A2A / ACP / Skills 各解决什么，怎么协作。写一个可复用 Skill。
依赖：Stage 3
课时：6-8h

L5.1 四层协议体系

协议解决的问题类比
MCPAgent 连接外部工具和数据源JDBC
A2AAgent 间发现、通信、协作微服务服务发现 + RPC
ACPAgent 与宿主应用的统一接口Servlet API
Skill可复用流程知识包Spring Boot Starter

L5.2 Skill vs Tool vs Prompt

  • Tool：可调用接口（函数级）
  • Prompt：一次性指令（无结构）
  • Skill：可发现、可版本化、可分发的能力包。触发条件 + 步骤 + 脚本 + 验收标准
"MCP 和 A2A 有什么区别？什么时候用 MCP，什么时候用 A2A？"

L5.3 MCP 深入

  • MCP Server 暴露 tools / resources / prompts
  • 传输层：stdio / SSE / Streamable HTTP
  • 设计一个最小 MCP Server

L5.4 A2A 协议

  • Agent Card：能力描述（类比 OpenAPI spec）
  • Task 生命周期：submitted -> working -> completed / failed
  • 消息格式：Message -> Part

L5.5 定向阅读：Skills 文件结构

  • 阅读 Claude Code Skills 的文件结构和触发机制
  • 阅读 OpenClaw Skills 的加载、作用域和安全边界
  • 阅读 Claude Agent Skills（agent 级 skill 设计）
  • 阅读 Claude Code Agent SDK Skills（SDK 集成 skill）

L5.6 实践：写一个 Skill

  • 最小 SKILL.md：name / description / when / steps / validation
  • 加脚本或模板文件
  • 写 smoke test
Claude Code Skills · Claude Agent Skills · Agent SDK Skills · OpenClaw Skills · MCP · A2A · ACP
提交可复用 Skill（SKILL.md + 脚本 + smoke test）。

P5 多 Agent 协调（Stage 4）

目标：设计小型多 Agent 系统，说清单 vs 多 Agent 取舍。
依赖：Stage 3
课时：6-8h

L4.1 常见模式

  • Supervisor：主 agent 分配任务，汇总结果
  • Router：按输入类型路由到专业 agent
  • Orchestrator-Workers：编排器拆分，worker 并行
  • Pipeline：research -> write -> review -> revise
  • Council：多 agent 独立回答，投票综合

L4.2 协调机制

  • 职责边界、输入输出 schema、停止条件
  • Agent 间通信：共享状态 vs 消息传递 vs blackboard

L4.3 常见问题

  • 循环和争论：加最大轮次 + 投票机制
  • 任务漂移：加目标检查点
  • 上下文膨胀：上下文隔离 + 摘要传递
  • 什么时候单 agent 更好
"什么场景选 multi-agent？主要工程挑战是什么？"
Claude Code Subagents · Claude Code Hooks · Google ADK · A2A Protocol · ACP
小型多 agent 系统：research -> write -> review -> revise。每个 agent 有输入输出 schema，有最大轮次限制。

P6 评测、可观测性与安全（Stage 7）

目标：设计 eval 表格，理解 trace 分析，知道常见安全风险。
依赖：Stage 1-3
课时：6-8h

L7.1 Agent Evaluation

  • 固定测试集 vs LLM-as-judge
  • 指标：成功率 / 准确率 / 工具调用次数 / 成本 / 延迟
  • 回归测试：prompt/工具改动后能力是否退化

L7.2 Observability / Trace

  • 定位失败：prompt / 工具 / 检索 / 模型 / 状态管理
  • Trace 内容：输入 / LLM 输出 / 工具调用 / 执行时间 / token
  • 工具：LangSmith / Phoenix / 自建

L7.3 安全

  • Prompt Injection：输入校验、指令分离、权限限制
  • Data Exfiltration：输出审查、网络隔离
  • Tool Abuse：权限分级、人工确认
  • 人工确认机制：高风险操作必须确认
"怎么评估 agent 效果？怎么定位 agent bug？怎么防止 prompt injection？"
OpenAI Evals · OpenAI Agent platform · LangSmith · AgentBench · SWE-bench
agent eval 表格：20+ 任务 / 期望结果 / 实际结果 / 失败分类。

P7 交付真实 Agent 项目（Stage 8）

目标：交付别人能 clone 下来跑的 agent 项目。
依赖：Stage 1-7
课时：12-16h

L8.1 项目要素

  • 明确用户、任务、成功标准
  • 日志、trace、重试、超时、成本上限
  • 权限边界和人工确认
  • 部署方式：CLI / Web / Slack bot / GitHub Action
  • README：运行、配置、扩展、限制

L8.2 方向

  • Coding Review Agent：diff -> 风险排序 -> 测试建议
  • PDF QA Agent：上传 -> RAG -> 引用式回答
  • Research Agent：主题 -> 搜索 -> 筛选 -> 报告
  • Multi-Agent Writer：planner -> writer -> reviewer
GitHub 仓库：README 完整 + eval 表格 + 完整 trace。别人 clone 后能跑起来。

Learning Principles（贯穿全课程）

  • Build first, then read deeper — 先动手再深读
  • Prefer small reliable agents over impressive demos — 小而可靠胜过大而花哨
  • Use tools with strict schemas — 工具用严格 schema
  • Add evals before you add more agents — 加 agent 前先加 eval
  • Trace every important run — 每次重要运行都 trace
  • Treat multi-agent as a coordination problem — 多 agent 是协调问题
  • Keep humans in the loop for risky actions — 危险操作保留人工确认
  • Respect platform rules, copyrights, and data access boundaries — 尊重平台规则
